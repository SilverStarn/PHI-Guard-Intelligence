# Technical Brief

![Technical architecture](assets/technical-architecture.svg)

## Purpose

PHI Guard Intelligence is a full-stack MVP for healthcare data risk intelligence. It demonstrates how a product can discover potential PHI risk, reason about lineage and access, explain findings, and produce remediation guidance while keeping raw PHI out of the durable result model.

## Architecture Summary

The system has four main layers:

1. Scanner and classifiers.
2. Risk rule engine.
3. API and metadata store.
4. React graph/reporting UI.

The scanner can run in demo mode, browser-upload mock mode, local file-agent mode, and PostgreSQL metadata mode. The intended production pattern is scanner-agent first: classify near the source, emit a sanitized package, and submit only metadata, labels, masked shapes, confidence, and finding evidence.

## Data Inputs

Supported local and mock inputs:

- CSV / TSV
- JSON / JSONL / NDJSON
- XML
- SQLite
- SQL DDL
- log and text files
- ZIP archives
- FHIR bundles
- dbt manifests
- PostgreSQL metadata scans

MySQL, SQL Server, S3, Azure Blob, and GCS are represented in the connector contract as future adapters.

## Classification Model

The scanner classifies columns and text patterns into labels such as:

- `DIRECT_IDENTIFIER`
- `QUASI_IDENTIFIER`
- `HEALTH_CONTEXT`
- `PAYMENT_CONTEXT`
- `FREE_TEXT_PHI_RISK`
- `LINKABLE_KEY`
- `DEIDENTIFICATION_BLOCKER`
- `ACCESS_CONTROL_GAP`
- `LINEAGE_PROPAGATED_RISK`
- `AI_EXPOSURE_RISK`

Classification uses schema names, data types, and masked sample patterns. Evidence is stored as safe snippets, confidence, and metadata, not raw values.

## Risk Reasoning

Findings are generated by deterministic rules for:

- Direct identifiers combined with health/payment context.
- Free-text PHI risk in patient-linked tables.
- Embedded PHI in logs, comments, prompts, or operational text.
- Sensitive mental-health context linked to identifier signals.
- Identifier flow into analytics, exports, dashboards, campaigns, or AI workflows.
- Broad role access to combined identifier and clinical/payment context.
- De-identification blockers in analytics/export tables.
- Joinability risk between identifiers and health/payment tables.
- Audit gaps for high-risk tables.
- Potential PHI exposure in AI workflow tables.

Risk scores combine impact, likelihood, exposure, control gap, and confidence, then apply rule-specific boosts. Findings include blast radius, control mapping, remediation summary, recommended steps, and human-review notes.

## Metadata Store

The durable target is PostgreSQL. The schema models:

- organizations, workspaces, users, roles, user_roles
- scanner_agents and agent_tokens
- data_sources and scan_runs
- assets, classifications, lineage_edges, access_grants
- findings and finding_evidence
- remediation_tasks
- audit_events, secrets, retention_policies

Local development can also store scan and audit summaries in `.phi_guard/metadata-store.jsonl`.

## Privacy And Security Posture

![Regulatory evidence loop](assets/regulatory-evidence-loop.svg)

Current guardrails:

- Demo data is synthetic.
- Browser upload mode is mock/local only.
- Scanner-agent packages include privacy metadata.
- API rejects packages with raw-retention markers set to true.
- Finding evidence stores `raw_value_stored=false`.
- Upload policy documents production controls that are not yet fully implemented.
- Product language avoids certification or legal-advice claims.

Production hardening path:

- Tenant auth and authorization.
- BAA workflow and subprocessor inventory.
- Encrypted object storage with short raw-upload TTL.
- Malware scan and isolated parser workers.
- Per-client KMS keys.
- Immutable audit logging.
- Dependency/container scanning and vulnerability management.
- Penetration testing and secure SDLC process.

## Frontend

The React/Vite UI includes:

- Dashboard metrics.
- Data risk graph with React Flow.
- Finding detail view.
- Remediation backlog.
- De-identification heatmap.
- Access matrix.
- Report view.
- Import/upload workflow and scan history.

The UI is intentionally operational rather than marketing-first. It presents a usable risk workbench as the first screen.
